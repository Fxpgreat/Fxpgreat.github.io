多线程编程与同步异步思考

从线程开始讲起，首先，如果一个程序只有一个线程，效率是很低的。
比如说，有一个函数要读写硬盘，这个耗时相对较长，只有一个执行流的话，只能干等。

如果有多条线程，可以将耗时任务交给线程 B，主线程 A 继续向下运行。
那么 B 运行好了怎么办呢？一种解决方案是 回调函数：将函数作为参数传给线程中的函数执行。

  • 如果主线程不在意结果 → 异步回调
  • 如果主线程在意结果 → 通知主线程接手结果

可以注意到，存在 同步 和 异步 两种情况：

  • 同步：有等待、依赖的情况，例如线程 A 依赖线程 B，要等 B 执行完才能执行 A
  • 异步：无等待、无依赖，各干各的。
    以前的单线程编程就是一路执行下去，遇到函数就要等它执行完。

还有两个概念：阻塞 与 非阻塞：

  • 阻塞：停下当前线程
  • 非阻塞：不停下当前线程

这四个词可以组合：

  • 阻塞式同步：A 线程停下，等待 B 线程完成后继续执行
  • 非阻塞同步：A 线程不停下，但一直询问 B 是否完成（资源浪费）
  • 阻塞式异步：A 没有完全依赖 B，但选择在 B 执行时主动暂停，B 完成后继续
  • 非阻塞异步：A 不关心 B，各干各的

先前看待同步异步不够精准，再复盘一下：

  • 同步/异步定义多种解释
    之前关注的更多是“依赖与等待”：
  • 同步 → 主线程阻塞调用某个函数，需要停下往下执行等待返回
  • 异步 → 调用后立马返回，即非阻塞调用

在 I/O 过程中：

  • 异步 → I/O 函数由内核完成
  • 同步 → I/O 函数由用户完成

更深入：

  • I/O 事件分为就绪与完成
  • 如果内核通知用户就绪，但完成需要用户 → 同步
  • 如果内核通知用户完成 → 异步

在通用编程中，从程序员视角：

  • 更关注当前线程是否依赖其他线程（是否阻塞）
  • 导致认知偏差，其实是定义冲突造成的

异步 I/O 发展演变：

  • 异步 I/O 不成熟时 → 用非阻塞调用 + 用户 I/O 模拟异步
  • 异步 I/O 成熟后 → 内核直接完成 I/O
  • 仍存在概念惯性，导致书本或文章中对同步/异步的描述不一致

还有一种情况：

  • 数据就绪后通知用户，用户处理完成再由内核进行 I/O → 异步 I/O
  • 核心关注点 → I/O 是否阻塞了线程

在并发模式中：

  • 同步 → 按顺序执行代码
  • 异步 → 程序执行依赖系统事件（如中断、信号）驱动

总结：
文章保留了当时的思考过程。
异步 I/O 的重点是 I/O 本身是异步的，线程是否阻塞只是程序员视角的体现。
理解这一点，有助于在网站上还原原始思考，而不是一味追求“正确”定义。

设计模式概览

设计模式的核心价值不在于发明新技术，而在于对已有技术的最佳实践进行命名、结构化描述和场景化指导。一般都分内部实现端和外部客户端：客户端只关心调用接口产生结果，内部只关心如何实现过程。

创造型模式都有抽象与具体的区别。

创造型模式

注重创造对象的机制

  • 工厂模式：解耦创建者与具体产品，当要创建新产品时只需要编写新的类接口
  • 抽象工厂：在工厂模式基础上，由一个抽象的族长管理族内接口
  • 建造者模式：通过指挥者唤起建造者，建造者按步骤完成具体部件的构建，步骤由指挥者定制
  • 原型模式：通过具体子原型类执行克隆返回一个一模一样的对象，可结合工厂模式
  • 单例模式：隐藏构造，初次通过全局静态函数创建后无法再创建，只返回已创建对象

结构型模式

注重对象和类的组织结构

  • 适配器模式：设计一个类内部维护被利用类的对象，帮助一个类使用被利用类的函数接口
  • 桥接模式：将一个大类的各个维度拆分为几个小类，并在内部相互关联，类似桥
  • 组合模式：类似树结构，树根是抽象类设计顶层功能，树枝负责与树叶（子节点）沟通，树叶实现具体功能
  • 装饰器模式：在基本装饰类的基础上用新的类添加动态新接口，不修改原类结构，像穿脱衣服
  • 外观模式：当功能类结构复杂、函数繁琐时，用一个对外类实现调用内部接口
  • 享元模式：正常创建多个对象时，每个对象的信息都会保存在容器中，但享元会共享已有信息
  • 代理模式：当真正功能类较大时，用代理类虚构相同接口去实际调用功能

行为型模式

负责对象间的沟通和职责委派

  • 责任链模式：将任务交给链条上的对象按顺序处理，每个对象指派下一个对象，可随时停止
  • 命令模式：调用类发送命令，用命令类封装指令，简单命令直接执行，复杂命令调用接收类接口执行
  • 迭代器模式：容器类保存数据，迭代器实现操作模板，容器通过迭代器读取数据，实现解耦
  • 中介者模式：取消功能类之间的直接通信，通过中介类完成操作通知和接口调用
  • 备忘录模式：备忘录类保存原发器对象的某时刻信息，由原发器提供接口复原或保存状态，负责人类统一管理
  • 观察者模式：被观察者类维护观察者容器，状态更新时依次调用观察者状态更新函数，实现实时监听
  • 状态模式：上下文类维护状态对象，状态对象由状态类产生，接口相同但功能不同，实现上下文状态转换
  • 策略模式：上下文类维护策略对象，策略对象由实现统一接口的具体策略类产生，算法不同，客户端可主动切换
  • 模板方法模式：将程序执行步骤拆分为函数，在抽象模板类声明并让子类实现，同时声明模板方法规定执行顺序
  • 访问者模式：将被访问对象与访问者分离，被作用元素作为参数传给访问方法，针对不同元素设置方法，访问者接口统一，实现不修改元素类结构

库文件与底层编译原理

先接触的库文件，库文件其实就是打包好的目标文件，里面有很多代码可供调用，然后我注意到是否必须要有头文件，头文件可供使用者查看库文件内的接口，但是我疑惑是否必须，按下不表，后来手动实现了一个静态库，其实就是先生成了目标文件，然后将它们合成为一个库文件，这一步也是由一个工具来执行的

然后重点来了，从更底层融会贯通，首先，CPU不懂抽象的东西，它只知道搬数据和计算，这些就体现在晶体管的开关，总体来看就是一堆数据的变化，那自然它只知道根据二进制来克制晶体管，这便是机器语言，只有机器能读的懂，后面抽象出了汇编，通过汇编器可以将汇编翻译为机器语言，但是不够简单抽象，太过一板一眼了，所以就有了高级语言，C＋＋就是一个编译型的高级语言，编译器会读取所有符号，然后形成语法树（AST），然后开始翻译其中的符号，遇到无法直接翻译的，它会注明出来以及还会给出全局的符号，最后产生也就是产生符号表，最后会有代码区数据区以及符号表，这就组成了目标文件，所以我说目标文件像积木，每个目标文件之间就像付出和索取，那么怎么将其组合在一起呢，这就是链接器的工作了，根据每个目标文件的符号表，总体检查一下每个需求有没有被满足，以及有没有重复 ，这就解释了我之前用vs时同一个项目里的多个源文件定义了同一个变量时为什么无法通过，因为ide把它们一锅炖全拿去编译链接了，那回答之前的问题，头文件到底是不是必须，回答是否，因为头文件是通过include指令被预处理器展开，所以头文件跟直接写入cpp文件没有什么区别，这也解释了为什么头文件路径很重要，因为要根据路径找到头文件，将内容复制到源文件里，那没有头文件也无所谓，可以一锅炖，但是不美观，不利于维护，而且制作库文件时，有头文件可以给使用者参考调用。所以库文件没什么大不了的，就是个目标文件集合，最后链接时满足别人的需求，静态库就是直接合入可执行文件，独立性强，动态库就是在可执行文件里留下库文件的标签，在使用时去调用，为什么可以这样？回到本质，因为所有东西归根到底都是在执行机器指令，那么可执行文件实际上也是一堆机器语言，所以在使用动态库时再去读取库的机器语言，没有问题。所以打开软件的文件夹，发现里面很多dll库文件，那这时候就注意到了，既然库文件是机器代码，那么是不是可以跨语言调用，实际上有可能可以的，但是要考虑不同语言最后的处理方式，这又设计到不同语言到底有什么区别了，归根到底，在编译型语言来说，区别就是它们到底最后翻译出什么汇编代码，因为最后是汇编出来的机器语言操控机器，比如c＋＋支持函数重载，这是怎么实现的呢，在于开发者怎么用汇编实现这一功能，那便是名称修饰，具体不说了，也就是汇编的跳转，跳转的函数不同，比如c写个add，函数在汇编里依然叫add，那只能用一个这种名称的函数，但是c＋＋，就将函数名改成其他东西，以区别重载函数，也就是名称修饰，这便是为什么不一定可以跨语言调库，因为细节是不同的

然后补充构建的知识，之前存在误区，其实构建的产物不一定是可执行文件，也可以是库文件之类的，这便是为什么可以构建出库文件，以及可以用cmake产生库文件，逻辑链条是，cmake读取cmakelists，这里是用比较简单抽象的指令告诉cmake你想干嘛，比如add-excutable就是使用源文件产生可执行文件，add-library就是产生库文件，然后cmake根据你的指令去调用生成器产生构建文件，构建文件就是一个指令合集，构建工具读取构建文件代替你手动操作指令去完成一系列操作，OK

做了一道dp，记住了，dp一定要有个过程，比如从0开始循环，一步步叠上去，以及二分，不要太纠结二分这个词，其实就是偷懒，从中间开始找，看看答案在哪一边，然后继续偷懒。

梦开始的地方 2025.1.30

这十多天学到的，一开始写了个机房预约系统，复习了一下C＋＋语法，还有git，git就是一个备份器，git仓库就是在本地文件夹创建一个git文件，里面有git执行所需要的文件，还有，为什么一些程序需要在终端打开，实际上就是没有图形化界面的程序其实都是在命令行打开，但是不可能做到随意打开，因为电脑不认识，所以要打开官方的入口创建一个虚拟机，里面继承了git所需的环境，然后就可以用git了，要记住。远程仓库其实就是一个云盘，将本地仓库在云端保存，不是默认实时的，需要本地自己推送到云端，然后本地有个分支指向云端的变化。因为云端一般也可以修改内容，推进分支。

后面搞了一下vscode，配置运行环境，理了一下IDE，编译器，可运行的关系，原来vscode不安装拓展的话只是一个好看一点的记事本，安装了C＋＋拓展之后，可以装载编译器，报错，自动填充等等功能，他需要用户写tasks，launch等json文件来指导自己如何去操作编译器，而vs比较智能，不用这样自己写，程序会自动生成，虽然vscode可以手动点出默认配置，然后ide根据这些文件去调用编译器 将代码转为可执行文件，每个ide都有自己的办法，devc＋＋就是自己内置了一个轻量化编译器。
