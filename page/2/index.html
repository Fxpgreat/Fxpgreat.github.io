LRU算法

本题为力扣 146. LRU 缓存：
请你设计并实现一个满足 LRU (最近最少使用) 缓存 约束的数据结构。

题目要求

实现 LRUCache 类：

  • LRUCache(int capacity)：以正整数作为容量 capacity 初始化 LRU 缓存。
  • int get(int key)：如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1。
  • void put(int key, int value)：如果关键字 key 已经存在，则变更其数据值 value；如果不存在，则向缓存中插入该组 key-value 。
    如果插入操作导致关键字数量超过 capacity ，则应该逐出最久未使用的关键字。

要求：函数 get 和 put 必须以 O(1) 的平均时间复杂度运行。

解题思路

分为三步：

  1. 构造函数
    初始化 LRUCache 类，维护好容量即可。

  2. 核心思想
    整个算法就是实现两个主要操作：get 和 put。

    • 只要有数据被访问（无论是读还是写），都属于“最近使用”，放到链表头。

  3. 具体操作

    • GET：查找 key → value，如果没找到返回 -1；如果找到了，移动该节点到链表头。
    • PUT：
      • 如果 key 存在，更新 value 并移动到链表头。
      • 如果 key 不存在，判断容量是否已满：
        • 若已满，删除链表尾部节点（最久未使用）。
        • 插入新的 (key,value) 到链表头。

代码实现 (C++)

#include <list>
#include <unordered_map>
using namespace std;

class LRUCache {
public:
    LRUCache(int capacity) {
        this->capacity = capacity;
    }
    
    int get(int key) {
        auto it = map.find(key);
        if (it == map.end()) {
            return -1;
        }
        // 移动到链表头
        cache.splice(cache.begin(), cache, it->second);
        return it->second->second;
    }
    
    void put(int key, int value) {
        auto it = map.find(key);
        if (it != map.end()) {
            // 更新 value，并移动到链表头
            it->second->second = value;
            cache.splice(cache.begin(), cache, it->second);
        } else {
            // 容量已满，删除尾部
            if (cache.size() == capacity) {
                auto del = cache.back();
                map.erase(del.first);
                cache.pop_back();
            }
            // 插入新节点到头部
            cache.emplace_front(key, value);
            map[key] = cache.begin();
        }
    }

private:
    int capacity;
    list<pair<int,int>> cache; // 双向链表：头=最近使用，尾=最久未使用
    unordered_map<int, list<pair<int,int>>::iterator> map; // 哈希表：key -> 链表节点
};

探究docker

容器技术的发展与启示

总的来说，docker就是轻量化的虚拟机，选取了进程运行必须依赖的操作系统模块，进行项目部署。

容器出现之前：部署的痛点

在容器（以 Docker 为代表的技术）出现之前，程序的部署往往有两类模式：

1. 直接部署

  • 方式：将程序直接部署到机器上运行，使用 systemd 等守护进程管理业务程序。
  • 优点：方便查看程序运行情况，支持手动/自动化启停。
  • 缺点：
    • 仅管理了业务程序本身，不管理依赖。
    • 新增机器时需手动安装依赖，依赖复杂且多版本共存困难。
    • 多个程序在同一台机器上可能发生依赖冲突（如 A 需要 X v1.0，而 B 需要 X v2.0，二者无法共存）。

总结：依赖安装简单，但依赖冲突和版本差异导致部署效率低、稳定性差，运维人员和开发者都很痛苦。

2. 虚拟机隔离

  • 思路：将程序与依赖环境完全绑定，作为独立单元（虚拟机镜像）进行管理。
  • 优点：
    • 彻底解决依赖冲突问题。
  • 缺点：
    • 镜像庞大：存储和传输成本高。
    • 启停缓慢：需要完整的操作系统启动。
    • 性能损耗大：多套内核进程导致资源浪费。

虚拟机解决了依赖冲突，但引入了新的性能与效率问题。

容器的基石：轻量化虚拟化技术

容器的目标是在解决依赖管理的同时，克服虚拟机的缺陷：镜像过大、启停缓慢、性能损耗。

1. 解决镜像过大：UnionFS

  • 虚拟机镜像庞大原因：每个镜像都包含操作系统的所有文件，存在大量冗余。
  • 优化思路：相同文件只保存一份，不同镜像共享，通过 Copy-on-Write 机制解决修改冲突。
  • 实现技术：联合文件系统（UnionFS，2004 年出现）。

2. 解决启停缓慢：Namespace + Cgroups

  • 问题：虚拟机启停包含完整操作系统开销。
  • 解决：利用 Linux 的 Namespace 和 Cgroups 实现轻量级隔离：
    • Namespace：隔离文件系统、进程视图、通信、网络、用户权限。
    • Cgroups：限制与分配 CPU、内存等硬件资源，避免资源抢占。
  • 代表：LXC（Linux Containers，2008 年成型）。

3. 解决性能损耗：共享内核

  • 虚拟机的损耗：多套操作系统导致重复进程与资源消耗。
  • 容器的优化：所有容器共享宿主机内核，避免冗余内核进程。
  • 结果：显著降低运行损耗，性能接近原生。

Docker 的出现

虽然容器相关技术（UnionFS、LXC）已经出现，但 缺乏完整的体系化方案。

为什么是 Docker？

  • 整合核心技术：基于 LXC 和 UnionFS，结合镜像机制。
  • 标准化交付：提供完整的打包（Dockerfile）、版本管理（Image）、分发（Registry）、运行管理（Runtime）。
  • 易用性极高：满足了开发者和运维的需求，实现 “Build once, Run anywhere”。
  • 命名寓意：Docker = 集装箱 → 软件交付标准化。
  • 开源传播：免费、开源，迅速成为容器领域的事实标准。

我们能学到什么？

1. 面向需求而不是面向技术

  • 真正有价值的产品解决的是 普遍痛点，而不是单纯追求技术上的极致。

2. 技术领域的组合创新

  • Docker 的成功不是凭空创造，而是整合了已有的底层技术（UnionFS、LXC）。
  • 要求工程师熟悉底层能力，能在合适时机进行组合。

3. 渐进式、问题驱动

  • 技术演进是一个渐进的过程：
    • 直接部署 → 虚拟机 → 容器。
  • 每一步都是为了解决新的问题，而不是一蹴而就。

4. 避开定势思维

  • 直接部署：执着于依赖管理 → 失败。
  • 虚拟机：执着于优化虚拟机 → 难以突破。
  • 容器：换思路 → 通过系统底层能力实现轻量化隔离。

启示：新的、更优秀的方案往往不是旧方案的改进，而是 根本性的思维变革。

docker如何使用

1. 编写dockerfile

在项目根目录创建一个dockerfile，在里面描述程序运行所需的环境（操作系统，依赖库，运行命令等）

2. 构建镜像

docker会读取dockerfile，逐层执行，生成一个镜像，每条指令对应镜像的一层（layer），方便使用

3.运行容器

容器 = 镜像＋可写层

查看容器

查看容器：docker ps
停止容器：docker stop <id>
查看容器：docker logs <id>
进入容器：docker exe -it <id> /bin/bash
发布容器（将镜像推送到远程仓库）：docker push name

链表归并排序

归并排序也就是分治思想，时间复杂度为O（nlogn），且属于稳定排序

流程：

  1. 分：递归地把数组分成两半，直到每个子数组只有一个元素（天然有序），其实也就是分到符合条件为止，在这里也就是有序。

  2. 治：把两个有序子数组 合并 成一个更大的有序数组，也就是将已经符合条件的部分进行合并。

  3. 分解：
    [4, 2, 1, 3]
    → [4, 2] [1, 3]
    → [4] [2] [1] [3]

  4. 合并：
    [4] + [2] → [2, 4]
    [1] + [3] → [1, 3]
    [2, 4] + [1, 3] → [1, 2, 3, 4]

CISC 与 RISC 架构

复杂指令集（CISC）

CISC 的特点是一条指令可以完成多个动作。举例：走路，本来要“迈左腿 → 迈右腿”，CISC 可能只用一条指令“走路”即可。

  • 问题：CISC 指令执行通常是硬连接，由电路直接完成，不便于代码移植和修改。
  • 解决方案：微代码（Microcode）
    • 将 CISC 指令拆解为更简单的微操作
    • 微代码存储在 ROM（只读寄存器）中
    • 修改 CISC 时，只需调整微代码即可
    • 缺点：修改微代码成本高，消耗晶体管

简单指令集（RISC）

RISC 指令集精简、底层，功能更基础：

  • 每条指令运行时间几乎相同，完美契合流水线技术
  • 编译器对代码控制力强
  • 市场上 RISC 一度占据优势

CISC 与 RISC 的市场对比

  • CISC 通过微操作提升底层执行效率
  • X86 架构（CISC）凭借市场基础击退 RISC（ARM 架构）
  • 移动互联网时代：Windows + Intel（Wintel 联盟）应对 ARM 仓促，未能击败 ARM
  • 移动市场被 ARM 架构占据
  • 苹果 M 系列芯片让 ARM 架构重回电脑市场

CPU原理与性能优化

CPU 的基本原理

计算机中最重要的部件是二极管，本质是一个开关——电流通过或不通过。通过组合与、或、非电路，可以实现任意计算（逻辑完备），还可以实现记忆电路。

  • 记忆电路：储存一个比特位
  • 寄存器：多个比特位组成
  • 内存：给每个比特位添加编号

通过电路的组建，CPU 的两大核心功能——计算与存储——得以实现。

CPU 不需要掌握所有功能，只需实现基本功能即可。程序员通过软件指导 CPU 的基本操作，从而完成复杂操作。这些基本功能就是机器指令，统称指令集，它是软件与硬件的交汇点。

数字的表达与计算

计算机只识别 0 和 1，以及基本计算操作。数字的具体意义由人类赋予：

  • 一个数字可以是变量值
  • 可以是内存地址
  • 可以是文件描述符
    无需纠结原码、补码、反码等细节，它们只是人类定义的表示方法。

CPU 性能优化

  1. 流水线技术

    • 指令执行分为取指、译码、执行、回写
    • 单步执行效率低，因为其他硬件单元空闲
    • 流水线让每个单元只做特定操作，提高吞吐量

  2. 分支预测

    • 条件语句可能阻塞流水线
    • CPU 猜测分支进行下一步计算
    • 若猜测正确，继续执行；若错误，回滚重做

内存池与内存申请机制

当程序需要申请内存时，一般会使用标准内存申请器（malloc 等），这是通用方案，但效率不高。内存池是针对特定场景设计的高效内存申请机制，只服务于特定模型。

简单内存池

  • 如果程序频繁申请同一个对象，可以预先创建一批对象
  • 当需要使用时直接调用，避免频繁申请和释放

复杂内存池

  • 支持各种大小内存的分配
  • 通过链表串联各块内存，用指针标记闲置内存供程序使用

可以看出，内存池是对 malloc 的抽象，而 malloc 又是对底层系统调用申请内存的抽象。

指针注意事项

  • 指针不能指向栈区数据，因为栈区内存随时可能被释放

补充知识

为什么硬盘不能当作内存使用？

  1. 硬盘读取速度远低于内存
  2. 虚拟内存容量有限（32 位系统进程虚拟内存只有 4GB），硬盘容量远大于需要
  3. 硬盘寻址粒度按块，而内存需要按字节访问

目前一些手机有内存扩容技术，将部分硬盘作为内存使用，但仍不能完全替代真实内存。

生产者-消费者模型与线程同步

生产者-消费者模型包含三个核心对象：生产者、消费者以及容器。
生产者将产生的对象放入容器供消费者使用，容器可以是链表、栈等数据结构。

当存在多个生产者和消费者时，会出现线程安全问题。使用互斥锁可以解决基本问题：

  • 在生产或消费操作时加锁
  • 执行完毕后释放锁

但是互斥锁存在局限：当容器满时，生产者仍然不断争抢执行权，浪费 CPU 资源。

条件变量

条件变量配合互斥锁使用，可以实现更智能的调度：

  • 当生产者生产后，通知消费者线程
  • 消费者通过 cond_wait 阻塞等待通知，接到通知后上锁并执行代码
  • 典型流程：生产者用 while 判断容器是否为空，循环执行 cond_wait，被唤醒后跳出循环执行后续操作

信号量

信号量是对线程可执行权的量化控制：

  • wait 操作：将信号量减 1，如果信号量为 0，则线程阻塞
  • post 操作：将信号量加 1，释放线程

信号量可以配合互斥锁在简单问题上使用，有效管理线程执行顺序和资源访问。

线程与同步

线程是在进程内部划分的执行流，具有轻量快捷的优点。一个进程内的多个线程共享一些资源，这就导致了线程最大的挑战——共享数据的安全问题。

先回顾线程的操作函数：

  • 创建
  • 终止
  • 连接（连接已终止的线程，回收其资源）
  • 线程分离（被分离的线程终止后自动回收资源）
  • 取消（在系统设置的取消点终止线程）

线程创建时可以传入属性，属性是一个结构体，需要初始化和设置。

线程同步

互斥锁

互斥锁通过阻塞实现同步：

  1. 创建锁
  2. 在临界区上锁，当线程进入临界区时抢占锁执行代码
  3. 其他线程到达临界区发现锁被占用，陷入阻塞
  4. 拥有锁的线程执行完临界区后释放锁，其他线程继续争抢执行权

互斥锁保证了共享数据的安全，但存在问题：

  • 死锁：忘记释放锁或重复加锁
  • 多线程多锁抢占：例如线程1持有锁a并尝试锁b，线程2持有锁b并尝试锁a，导致停滞

读写锁

互斥锁在临界区仅读操作时也会阻塞其他线程，影响效率。
读写锁可以区分读与写：

  • 加 读锁：允许其他线程同时读取数据
  • 加 写锁：阻止其他线程读写数据，写锁优先级最高

文件描述符

进程内核中独立维护一个 文件描述符表，从 0 开始递增：0、1、2 分别对应宏定义的 STDIN、STDOUT、STDERR。

当通过 open 打开一个文件（如文件 a）时，内核会在文件描述符表中寻找第一个空闲位置，将其定向到文件 a 的 文件表。程序员可以用一个变量获取这个描述符，例如 3，那么 3 就指向文件 a。

注意：文件描述符本质只是一个 int 数据，关键是程序如何理解这个 int。例如在 dup、dup2 中，传入的 int 会被解释为文件描述符，内核就会通过这个值找到文件表，再通过 i 节点定位文件的物理地址等信息。

一般的文件 IO 都是通过文件描述符操作的。

  • dup：在描述符表中再找一个空闲符，将其指向传入描述符所指向的文件表，然后返回给程序员，相当于复制了一个描述符。
  • dup2(a, b)：如果目标描述符 b 是空闲的，直接指向 a 的文件表；如果 b 已占用，会先 close 再指向 a 的文件表。这一步是 原子操作，不可分割中断。
