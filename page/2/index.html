CISC 与 RISC 架构

复杂指令集（CISC）

CISC 的特点是一条指令可以完成多个动作。举例：走路，本来要“迈左腿 → 迈右腿”，CISC 可能只用一条指令“走路”即可。

  • 问题：CISC 指令执行通常是硬连接，由电路直接完成，不便于代码移植和修改。
  • 解决方案：微代码（Microcode）
    • 将 CISC 指令拆解为更简单的微操作
    • 微代码存储在 ROM（只读寄存器）中
    • 修改 CISC 时，只需调整微代码即可
    • 缺点：修改微代码成本高，消耗晶体管

简单指令集（RISC）

RISC 指令集精简、底层，功能更基础：

  • 每条指令运行时间几乎相同，完美契合流水线技术
  • 编译器对代码控制力强
  • 市场上 RISC 一度占据优势

CISC 与 RISC 的市场对比

  • CISC 通过微操作提升底层执行效率
  • X86 架构（CISC）凭借市场基础击退 RISC（ARM 架构）
  • 移动互联网时代：Windows + Intel（Wintel 联盟）应对 ARM 仓促，未能击败 ARM
  • 移动市场被 ARM 架构占据
  • 苹果 M 系列芯片让 ARM 架构重回电脑市场

CPU原理与性能优化

CPU 的基本原理

计算机中最重要的部件是二极管，本质是一个开关——电流通过或不通过。通过组合与、或、非电路，可以实现任意计算（逻辑完备），还可以实现记忆电路。

  • 记忆电路：储存一个比特位
  • 寄存器：多个比特位组成
  • 内存：给每个比特位添加编号

通过电路的组建，CPU 的两大核心功能——计算与存储——得以实现。

CPU 不需要掌握所有功能，只需实现基本功能即可。程序员通过软件指导 CPU 的基本操作，从而完成复杂操作。这些基本功能就是机器指令，统称指令集，它是软件与硬件的交汇点。

数字的表达与计算

计算机只识别 0 和 1，以及基本计算操作。数字的具体意义由人类赋予：

  • 一个数字可以是变量值
  • 可以是内存地址
  • 可以是文件描述符
    无需纠结原码、补码、反码等细节，它们只是人类定义的表示方法。

CPU 性能优化

  1. 流水线技术

    • 指令执行分为取指、译码、执行、回写
    • 单步执行效率低，因为其他硬件单元空闲
    • 流水线让每个单元只做特定操作，提高吞吐量

  2. 分支预测

    • 条件语句可能阻塞流水线
    • CPU 猜测分支进行下一步计算
    • 若猜测正确，继续执行；若错误，回滚重做

内存池与内存申请机制

当程序需要申请内存时，一般会使用标准内存申请器（malloc 等），这是通用方案，但效率不高。内存池是针对特定场景设计的高效内存申请机制，只服务于特定模型。

简单内存池

  • 如果程序频繁申请同一个对象，可以预先创建一批对象
  • 当需要使用时直接调用，避免频繁申请和释放

复杂内存池

  • 支持各种大小内存的分配
  • 通过链表串联各块内存，用指针标记闲置内存供程序使用

可以看出，内存池是对 malloc 的抽象，而 malloc 又是对底层系统调用申请内存的抽象。

指针注意事项

  • 指针不能指向栈区数据，因为栈区内存随时可能被释放

补充知识

为什么硬盘不能当作内存使用？

  1. 硬盘读取速度远低于内存
  2. 虚拟内存容量有限（32 位系统进程虚拟内存只有 4GB），硬盘容量远大于需要
  3. 硬盘寻址粒度按块，而内存需要按字节访问

目前一些手机有内存扩容技术，将部分硬盘作为内存使用，但仍不能完全替代真实内存。

生产者-消费者模型与线程同步

生产者-消费者模型包含三个核心对象：生产者、消费者以及容器。
生产者将产生的对象放入容器供消费者使用，容器可以是链表、栈等数据结构。

当存在多个生产者和消费者时，会出现线程安全问题。使用互斥锁可以解决基本问题：

  • 在生产或消费操作时加锁
  • 执行完毕后释放锁

但是互斥锁存在局限：当容器满时，生产者仍然不断争抢执行权，浪费 CPU 资源。

条件变量

条件变量配合互斥锁使用，可以实现更智能的调度：

  • 当生产者生产后，通知消费者线程
  • 消费者通过 cond_wait 阻塞等待通知，接到通知后上锁并执行代码
  • 典型流程：生产者用 while 判断容器是否为空，循环执行 cond_wait，被唤醒后跳出循环执行后续操作

信号量

信号量是对线程可执行权的量化控制：

  • wait 操作：将信号量减 1，如果信号量为 0，则线程阻塞
  • post 操作：将信号量加 1，释放线程

信号量可以配合互斥锁在简单问题上使用，有效管理线程执行顺序和资源访问。

线程与同步

线程是在进程内部划分的执行流，具有轻量快捷的优点。一个进程内的多个线程共享一些资源，这就导致了线程最大的挑战——共享数据的安全问题。

先回顾线程的操作函数：

  • 创建
  • 终止
  • 连接（连接已终止的线程，回收其资源）
  • 线程分离（被分离的线程终止后自动回收资源）
  • 取消（在系统设置的取消点终止线程）

线程创建时可以传入属性，属性是一个结构体，需要初始化和设置。

线程同步

互斥锁

互斥锁通过阻塞实现同步：

  1. 创建锁
  2. 在临界区上锁，当线程进入临界区时抢占锁执行代码
  3. 其他线程到达临界区发现锁被占用，陷入阻塞
  4. 拥有锁的线程执行完临界区后释放锁，其他线程继续争抢执行权

互斥锁保证了共享数据的安全，但存在问题：

  • 死锁：忘记释放锁或重复加锁
  • 多线程多锁抢占：例如线程1持有锁a并尝试锁b，线程2持有锁b并尝试锁a，导致停滞

读写锁

互斥锁在临界区仅读操作时也会阻塞其他线程，影响效率。
读写锁可以区分读与写：

  • 加 读锁：允许其他线程同时读取数据
  • 加 写锁：阻止其他线程读写数据，写锁优先级最高

文件描述符

进程内核中独立维护一个 文件描述符表，从 0 开始递增：0、1、2 分别对应宏定义的 STDIN、STDOUT、STDERR。

当通过 open 打开一个文件（如文件 a）时，内核会在文件描述符表中寻找第一个空闲位置，将其定向到文件 a 的 文件表。程序员可以用一个变量获取这个描述符，例如 3，那么 3 就指向文件 a。

注意：文件描述符本质只是一个 int 数据，关键是程序如何理解这个 int。例如在 dup、dup2 中，传入的 int 会被解释为文件描述符，内核就会通过这个值找到文件表，再通过 i 节点定位文件的物理地址等信息。

一般的文件 IO 都是通过文件描述符操作的。

  • dup：在描述符表中再找一个空闲符，将其指向传入描述符所指向的文件表，然后返回给程序员，相当于复制了一个描述符。
  • dup2(a, b)：如果目标描述符 b 是空闲的，直接指向 a 的文件表；如果 b 已占用，会先 close 再指向 a 的文件表。这一步是 原子操作，不可分割中断。

终端、进程组与守护进程

首先，创建一个进程时，如果它是父进程，它便是一个进程组的组长。多个进程组可以存在于同一个会话中，此时父进程是会话的首个进程组。如果一个子进程申请创建会话，它会重新执行这些步骤。

也就是说，进程 → 进程组 → 会话，然后一个会话关联一个控制终端，指定一个前台进程与用户交互，进行输入输出。

因此，自然就有后台进程，不受控制终端影响。

守护进程是需要在后台静默长时间运行的进程，一般随系统启动而运行，随系统关闭而结束。

守护进程的创建过程：

  1. 建立一个子进程
  2. 创建一个新会话
  3. 关闭父进程
  4. 关闭标准输入、输出、错误文件描述符，并重定向到 /dev/null
  5. 选择性修改工作目录
  6. 实现业务逻辑

进程间通信

学习完进程之后知道，进程之间是相对独立的，那自然要知道进程之间如何通信，分为三种：管道、内存映射和信号。

管道通信

匿名管道：在内核区中开辟一个缓冲区来交流。具体就是一个进程写入数据，另一个进程读取数据，完成通信功能。匿名管道没有文件实体，只是一个缓冲区，用两个文件描述符，一个记录写端，一个记录读端。用 write 和 read 函数来实现通信。注意，管道的读写需要固定，只能读或者只能写，否则会混乱。

有名管道：通过一个文件实体（FIFO）来实现通信，通过该文件的文件描述符调用文件 IO 函数实现通信。若要实现双向通信，需要两个管道。

在实时聊天实现中，由于父子进程共享终端读写，聊天双方可以用两个进程分别实现读写，读用 read，写用 write，文件描述符通过 open 获得。

信号通信

信号是一种古老而简单的通信方式，当一个事件发生时，有进程向当前进程发送信号，当前进程便中断去处理。主要用途有两点：告知进程事件发生和强迫进程处理信号。

  • 信号简单，不能携带大量信息，优先级高。
  • alarm 定时器就是一定时间后发送信号。
  • signal 和 sigaction 可以捕捉指定信号，并用回调函数处理。
  • 信号集：每个进程 PCB 中列出的信号状态清单，可设置阻塞信号。

例如解决僵尸进程问题：当子进程结束时会向父进程发送 SIGCHLD 信号，父进程可以用 sigaction 捕捉信号，并执行 wait 回收子进程。

内存映射

内存映射将硬盘数据映射到内存，通过修改内存来同步硬盘。父子进程间通信可在子进程创建前建立映射并获得地址，通过读写共享内存实现通信。

  • 可实现文件拷贝：创建两个文件映射区，然后拷贝。
  • 父子进程可选择匿名映射，不创建文件实体。
  • 优点：安全性高，即便通信过程中进程崩溃，文件内容不会丢失。

共享内存（System V）

共享内存是另一种进程通信方式，开辟一块内存让多个进程同时访问。

  • 与内存映射不同，共享内存不涉及文件映射。
  • 速度快，适合少量多次通信。
  • 安全性不如内存映射，内存重置会丢失通信内容。
  • 一个进程不能独立删除共享内存，只能标记删除并脱离，所有进程脱离后自动删除。

多进程与进程状态

课程中再次学习了进程，果然学习深入了疑惑自然迎刃而解。现阶段对于进程认知：计算机分配给程序的一块内存，程序在这块内存上完成任务，是程序运行的实例化。这里有一些感想：计算机中的任务都不是天马行空的，每一个行为的背后都有数据和函数在支撑维护。

每个进程有一个内核用于管理进程和一些信息，比如 PCB（程序控制块），用于维护进程相关信息，如进程 ID。

进程关系都是链条结构的，一个进程一定是其父进程创建的。init 是初始进程，PID 为 1。用 fork 可以为当前进程创建一个子进程，子进程会将剩余的代码、全部数据、文件描述符同样复制到自己的进程内存中。请注意，一定避免误区：不是每个进程都要有自己的代码页，代码编辑只是程序员的工作，子进程自己就是一个完整的运行中程序，只不过大部分内容来源其父进程。fork 有一个提升效率的机制——写时拷贝（Copy-On-Write），在子进程实际修改数据之前，虚拟内存都实际指向父进程相关内容的物理内容。以及输入输出的三个文件描述符都是与父进程共享，这就是为什么父子进程可以共享一个终端页面。

那问题来了，难道子进程不能运行其他东西吗？可以的，用 exec 函数组，可以用其他可执行文件替换子进程，代码区、数据区、堆栈全部被替换，只留下一些代表这个进程的信息，就像“夺舍”了一样。

接着讲孤儿进程和僵尸进程：

  • 孤儿进程：父进程已经结束，但子进程还在运行，导致其没有父进程。这时 init 进程会接管子进程，等待它结束后回收资源。
  • 僵尸进程：子进程运行完毕，自己释放了大部分资源，留下内核区的 PCB，需要父进程来释放。但是此时父进程可能因阻塞等原因无法对子进程执行释放，导致子进程成为“没有灵魂”的僵尸。僵尸进程比较危险，只有其父进程才能处理它，也无法被杀死。

进程回收只能由父进程完成，可以用 wait 或 waitpid 完成，有一个传出参数 waitstat，可以用函数分析这个参数，获取子进程回收时的一些状态。

栈区与堆区解析

程序在内存中，也就是进程的几个区：内核区、栈区、共享区、堆区、全局数据区、代码区和常量区。其他几个区已经大致了解了，现在详细剖析一下栈与堆。

栈区

为什么需要栈区呢？因为函数调用时经常会调用其他函数，或者依赖于其他函数的结果。
例如：函数 a 要等 b 执行完才能执行，b 的前置又是 c，那么函数执行顺序是 c → b → a，这是先进后出的顺序，与栈完美契合。

所以在进程内存中开辟了一个栈区，为每个函数分配一个栈帧，可以理解为信息存档。

  • 返回地址：当 a 调用 b 时，需要知道执行完 b 后如何返回 a，返回地址就存储在 a 的栈帧中。
  • 参数与返回值：函数 a 将参数写入寄存器，b 直接调用，返回值写回寄存器。如果参数过多超过寄存器数量，则存入栈帧。
  • 局部变量：函数内创建的变量也存储在栈中。
  • 寄存器保护：函数 a 使用的寄存器交给 b 前，会先保存到栈中，避免被覆盖，返回时再恢复。

堆区

为什么需要堆区呢？因为栈的数据由操作系统自动创建和释放，程序员无法直接管理。

  • 手动管理：堆区允许程序员手动管理内存，用于跨线程或生命周期不确定的数据。
  • 分配与释放：使用 malloc 或 new 创建，返回指针，指向堆内存地址。
  • 内存管理：每块内存首尾有信息块记录大小，可形成类似双向链表结构。
  • 延迟映射：malloc 返回的虚拟内存不一定对应物理内存，只有真正使用时，操作系统会分配物理页。
  • 系统调用：当堆区满时，会向系统申请共享区，需要使用操作系统提供的接口（系统调用）。
  • 标准库：C++ 提供跨平台标准库接口，底层通过系统调用实现内存管理。

堆区机制比栈区简单，但需要程序员自行控制生命周期和大小。通过理解栈和堆，可以更深入地掌握内存管理和底层编程原理。
