1. 初识Kubernetes：解决认知困惑

1.1 问题的起源

作为一个接触Kubernetes的新手，最大的困惑来自于：

  • 侵入性太强：K8s似乎要改变整个应用的架构
  • 概念抽象：Pod、Service、Deployment等概念难以理解
  • 黑箱操作：不知道Pod是如何创建和运行的

1.2 认知转变的关键

核心理解：Kubernetes不是应用框架，而是基础设施抽象层。

1.3 模块化视角下的K8s

Kubernetes本质上是一个模块化的分布式系统：

控制平面模块：

  • kube-apiserver: API网关
  • etcd: 分布式配置存储
  • kube-scheduler: 资源调度器
  • kube-controller-manager: 状态控制器

工作节点模块：

  • kubelet: 节点代理
  • kube-proxy: 网络代理
  • 容器运行时: Docker/containerd

10. 实践总结与最佳实践

10.1 核心概念总结

Kubernetes的本质

  • 定位：分布式系统的基础设施抽象层
  • 作用：资源管理、调度、生命周期管理
  • 价值：自动化运维、标准化部署、服务治理

Pod的本质

  • 定义：Kubernetes的最小部署单元
  • 组成：一个或多个容器的集合
  • 特点：共享网络、存储、生命周期

调度机制

  • 过程：API Server → Scheduler → kubelet → 容器运行时
  • 结果：Pod运行在实际的物理机或虚拟机上
  • 控制：通过资源请求、节点选择器、亲和性等策略控制

10.2 设计原则和最佳实践

容器设计原则

  1. 优先使用单容器Pod：简单、清晰、易管理
  2. 多容器Pod的使用条件：
    • 容器间需要共享文件系统
    • 容器间需要共享网络（localhost通信）
    • 容器生命周期紧密耦合
    • 容器需要同时调度到同一节点

部署策略

  1. 渐进式迁移：

    • 第一步：容器化应用
    • 第二步：单Pod部署
    • 第三步：添加K8s特性
    • 第四步：完整的云原生架构

  2. 环境选择：

    • 开发环境：Minikube、Kind、Docker Desktop
    • 测试环境：kubeadm集群
    • 生产环境：云服务商托管或自建高可用集群

验证二叉搜索树

题目描述

本题为 力扣 98. 验证二叉搜索树。
给你一个二叉树的根节点 root ，判断其是否是一个有效的二叉搜索树。

有效二叉搜索树定义如下：

  1. 节点的左子树只包含 严格小于 当前节点的数。
  2. 节点的右子树只包含 严格大于 当前节点的数。
  3. 所有左子树和右子树自身必须也是二叉搜索树。

解题思路

因为本题涉及二叉树的遍历，通常使用 递归 来完成。
需要注意的一点是：
判断 BST 时，比较的是 整个子树 是否满足条件，而不是仅比较 左右子节点。

因此每次递归到一个节点时，需要检查它是否在 祖先节点传下来的约束范围 内。
即：递归时为每个节点维护一个区间 (min, max)。

递归规则如下：

  • 左子树递归时：
    helper(node.Left, min, node.Val)

  • 右子树递归时：
    helper(node.Right, node.Val, max)

为什么每次只更新一边？
因为约束是延续的：

  • 对左子树来说，最大值受当前节点的值限制；
  • 对右子树来说，最小值受当前节点的值限制。

代码实现（Go）

/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val   int
 *     Left  *TreeNode
 *     Right *TreeNode
 * }
 */
func isValidBST(root *TreeNode) bool {
    return helper(root, nil, nil)
}

func helper(node *TreeNode, min *int, max *int) bool {
    if node == nil {
        return true
    }
    if min != nil && node.Val <= *min {
        return false
    }
    if max != nil && node.Val >= *max {
        return false
    }
    return helper(node.Left, min, &node.Val) && helper(node.Right, &node.Val, max)
}

Go 的鸭子原则要求：类型要被视为接口，必须实现接口的所有方法。编译器会自动认定该类型实现了接口。接口不是对象，也不是类，只是一组方法签名的集合。带接收者的函数就是该类型的方法，所有方法合起来构成类型的方法集，且方法集是静态属性，编译器完全可知。在多态调用时，编译器会检查类型的方法集是否完全覆盖接口的方法集，否则报错。在内存层面，接口变量存储动态类型和具体值（或指针），方法调用时通过方法表查找具体类型的实现，从而实现多态。

LRU算法

本题为力扣 146. LRU 缓存：
请你设计并实现一个满足 LRU (最近最少使用) 缓存 约束的数据结构。

题目要求

实现 LRUCache 类：

  • LRUCache(int capacity)：以正整数作为容量 capacity 初始化 LRU 缓存。
  • int get(int key)：如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1。
  • void put(int key, int value)：如果关键字 key 已经存在，则变更其数据值 value；如果不存在，则向缓存中插入该组 key-value 。
    如果插入操作导致关键字数量超过 capacity ，则应该逐出最久未使用的关键字。

要求：函数 get 和 put 必须以 O(1) 的平均时间复杂度运行。

解题思路

分为三步：

  1. 构造函数
    初始化 LRUCache 类，维护好容量即可。

  2. 核心思想
    整个算法就是实现两个主要操作：get 和 put。

    • 只要有数据被访问（无论是读还是写），都属于“最近使用”，放到链表头。

  3. 具体操作

    • GET：查找 key → value，如果没找到返回 -1；如果找到了，移动该节点到链表头。
    • PUT：
      • 如果 key 存在，更新 value 并移动到链表头。
      • 如果 key 不存在，判断容量是否已满：
        • 若已满，删除链表尾部节点（最久未使用）。
        • 插入新的 (key,value) 到链表头。

代码实现 (C++)

#include <list>
#include <unordered_map>
using namespace std;

class LRUCache {
public:
    LRUCache(int capacity) {
        this->capacity = capacity;
    }
    
    int get(int key) {
        auto it = map.find(key);
        if (it == map.end()) {
            return -1;
        }
        // 移动到链表头
        cache.splice(cache.begin(), cache, it->second);
        return it->second->second;
    }
    
    void put(int key, int value) {
        auto it = map.find(key);
        if (it != map.end()) {
            // 更新 value，并移动到链表头
            it->second->second = value;
            cache.splice(cache.begin(), cache, it->second);
        } else {
            // 容量已满，删除尾部
            if (cache.size() == capacity) {
                auto del = cache.back();
                map.erase(del.first);
                cache.pop_back();
            }
            // 插入新节点到头部
            cache.emplace_front(key, value);
            map[key] = cache.begin();
        }
    }

private:
    int capacity;
    list<pair<int,int>> cache; // 双向链表：头=最近使用，尾=最久未使用
    unordered_map<int, list<pair<int,int>>::iterator> map; // 哈希表：key -> 链表节点
};

探究docker

容器技术的发展与启示

总的来说，docker就是轻量化的虚拟机，选取了进程运行必须依赖的操作系统模块，进行项目部署。

容器出现之前：部署的痛点

在容器（以 Docker 为代表的技术）出现之前，程序的部署往往有两类模式：

1. 直接部署

  • 方式：将程序直接部署到机器上运行，使用 systemd 等守护进程管理业务程序。
  • 优点：方便查看程序运行情况，支持手动/自动化启停。
  • 缺点：
    • 仅管理了业务程序本身，不管理依赖。
    • 新增机器时需手动安装依赖，依赖复杂且多版本共存困难。
    • 多个程序在同一台机器上可能发生依赖冲突（如 A 需要 X v1.0，而 B 需要 X v2.0，二者无法共存）。

总结：依赖安装简单，但依赖冲突和版本差异导致部署效率低、稳定性差，运维人员和开发者都很痛苦。

2. 虚拟机隔离

  • 思路：将程序与依赖环境完全绑定，作为独立单元（虚拟机镜像）进行管理。
  • 优点：
    • 彻底解决依赖冲突问题。
  • 缺点：
    • 镜像庞大：存储和传输成本高。
    • 启停缓慢：需要完整的操作系统启动。
    • 性能损耗大：多套内核进程导致资源浪费。

虚拟机解决了依赖冲突，但引入了新的性能与效率问题。

容器的基石：轻量化虚拟化技术

容器的目标是在解决依赖管理的同时，克服虚拟机的缺陷：镜像过大、启停缓慢、性能损耗。

1. 解决镜像过大：UnionFS

  • 虚拟机镜像庞大原因：每个镜像都包含操作系统的所有文件，存在大量冗余。
  • 优化思路：相同文件只保存一份，不同镜像共享，通过 Copy-on-Write 机制解决修改冲突。
  • 实现技术：联合文件系统（UnionFS，2004 年出现）。

2. 解决启停缓慢：Namespace + Cgroups

  • 问题：虚拟机启停包含完整操作系统开销。
  • 解决：利用 Linux 的 Namespace 和 Cgroups 实现轻量级隔离：
    • Namespace：隔离文件系统、进程视图、通信、网络、用户权限。
    • Cgroups：限制与分配 CPU、内存等硬件资源，避免资源抢占。
  • 代表：LXC（Linux Containers，2008 年成型）。

3. 解决性能损耗：共享内核

  • 虚拟机的损耗：多套操作系统导致重复进程与资源消耗。
  • 容器的优化：所有容器共享宿主机内核，避免冗余内核进程。
  • 结果：显著降低运行损耗，性能接近原生。

Docker 的出现

虽然容器相关技术（UnionFS、LXC）已经出现，但 缺乏完整的体系化方案。

为什么是 Docker？

  • 整合核心技术：基于 LXC 和 UnionFS，结合镜像机制。
  • 标准化交付：提供完整的打包（Dockerfile）、版本管理（Image）、分发（Registry）、运行管理（Runtime）。
  • 易用性极高：满足了开发者和运维的需求，实现 “Build once, Run anywhere”。
  • 命名寓意：Docker = 集装箱 → 软件交付标准化。
  • 开源传播：免费、开源，迅速成为容器领域的事实标准。

我们能学到什么？

1. 面向需求而不是面向技术

  • 真正有价值的产品解决的是 普遍痛点，而不是单纯追求技术上的极致。

2. 技术领域的组合创新

  • Docker 的成功不是凭空创造，而是整合了已有的底层技术（UnionFS、LXC）。
  • 要求工程师熟悉底层能力，能在合适时机进行组合。

3. 渐进式、问题驱动

  • 技术演进是一个渐进的过程：
    • 直接部署 → 虚拟机 → 容器。
  • 每一步都是为了解决新的问题，而不是一蹴而就。

4. 避开定势思维

  • 直接部署：执着于依赖管理 → 失败。
  • 虚拟机：执着于优化虚拟机 → 难以突破。
  • 容器：换思路 → 通过系统底层能力实现轻量化隔离。

启示：新的、更优秀的方案往往不是旧方案的改进，而是 根本性的思维变革。

docker如何使用

1. 编写dockerfile

在项目根目录创建一个dockerfile，在里面描述程序运行所需的环境（操作系统，依赖库，运行命令等）

2. 构建镜像

docker会读取dockerfile，逐层执行，生成一个镜像，每条指令对应镜像的一层（layer），方便使用

3.运行容器

容器 = 镜像＋可写层

查看容器

查看容器：docker ps
停止容器：docker stop <id>
查看容器：docker logs <id>
进入容器：docker exe -it <id> /bin/bash
发布容器（将镜像推送到远程仓库）：docker push name

链表归并排序

归并排序也就是分治思想，时间复杂度为O（nlogn），且属于稳定排序

流程：

  1. 分：递归地把数组分成两半，直到每个子数组只有一个元素（天然有序），其实也就是分到符合条件为止，在这里也就是有序。

  2. 治：把两个有序子数组 合并 成一个更大的有序数组，也就是将已经符合条件的部分进行合并。

  3. 分解：
    [4, 2, 1, 3]
    → [4, 2] [1, 3]
    → [4] [2] [1] [3]

  4. 合并：
    [4] + [2] → [2, 4]
    [1] + [3] → [1, 3]
    [2, 4] + [1, 3] → [1, 2, 3, 4]

CISC 与 RISC 架构

复杂指令集（CISC）

CISC 的特点是一条指令可以完成多个动作。举例：走路，本来要“迈左腿 → 迈右腿”，CISC 可能只用一条指令“走路”即可。

  • 问题：CISC 指令执行通常是硬连接，由电路直接完成，不便于代码移植和修改。
  • 解决方案：微代码（Microcode）
    • 将 CISC 指令拆解为更简单的微操作
    • 微代码存储在 ROM（只读寄存器）中
    • 修改 CISC 时，只需调整微代码即可
    • 缺点：修改微代码成本高，消耗晶体管

简单指令集（RISC）

RISC 指令集精简、底层，功能更基础：

  • 每条指令运行时间几乎相同，完美契合流水线技术
  • 编译器对代码控制力强
  • 市场上 RISC 一度占据优势

CISC 与 RISC 的市场对比

  • CISC 通过微操作提升底层执行效率
  • X86 架构（CISC）凭借市场基础击退 RISC（ARM 架构）
  • 移动互联网时代：Windows + Intel（Wintel 联盟）应对 ARM 仓促，未能击败 ARM
  • 移动市场被 ARM 架构占据
  • 苹果 M 系列芯片让 ARM 架构重回电脑市场

CPU原理与性能优化

CPU 的基本原理

计算机中最重要的部件是二极管，本质是一个开关——电流通过或不通过。通过组合与、或、非电路，可以实现任意计算（逻辑完备），还可以实现记忆电路。

  • 记忆电路：储存一个比特位
  • 寄存器：多个比特位组成
  • 内存：给每个比特位添加编号

通过电路的组建，CPU 的两大核心功能——计算与存储——得以实现。

CPU 不需要掌握所有功能，只需实现基本功能即可。程序员通过软件指导 CPU 的基本操作，从而完成复杂操作。这些基本功能就是机器指令，统称指令集，它是软件与硬件的交汇点。

数字的表达与计算

计算机只识别 0 和 1，以及基本计算操作。数字的具体意义由人类赋予：

  • 一个数字可以是变量值
  • 可以是内存地址
  • 可以是文件描述符
    无需纠结原码、补码、反码等细节，它们只是人类定义的表示方法。

CPU 性能优化

  1. 流水线技术

    • 指令执行分为取指、译码、执行、回写
    • 单步执行效率低，因为其他硬件单元空闲
    • 流水线让每个单元只做特定操作，提高吞吐量

  2. 分支预测

    • 条件语句可能阻塞流水线
    • CPU 猜测分支进行下一步计算
    • 若猜测正确，继续执行；若错误，回滚重做

内存池与内存申请机制

当程序需要申请内存时，一般会使用标准内存申请器（malloc 等），这是通用方案，但效率不高。内存池是针对特定场景设计的高效内存申请机制，只服务于特定模型。

简单内存池

  • 如果程序频繁申请同一个对象，可以预先创建一批对象
  • 当需要使用时直接调用，避免频繁申请和释放

复杂内存池

  • 支持各种大小内存的分配
  • 通过链表串联各块内存，用指针标记闲置内存供程序使用

可以看出，内存池是对 malloc 的抽象，而 malloc 又是对底层系统调用申请内存的抽象。

指针注意事项

  • 指针不能指向栈区数据，因为栈区内存随时可能被释放

补充知识

为什么硬盘不能当作内存使用？

  1. 硬盘读取速度远低于内存
  2. 虚拟内存容量有限（32 位系统进程虚拟内存只有 4GB），硬盘容量远大于需要
  3. 硬盘寻址粒度按块，而内存需要按字节访问

目前一些手机有内存扩容技术，将部分硬盘作为内存使用，但仍不能完全替代真实内存。
